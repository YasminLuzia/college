Um analista de sistemas tem um papel importantíssimo na construção de um software, pois ele é o responsável pelo levantamento de requisitos. Os requisitos fazem parte da grande documentação de um software e é o foco no momento.
Os requisitos de um software são importantes para documenta-lo, para mostrar aos clientes o trabalho que será feito e para guiar os responsáveis pelo processo de desenvolvimento. Existem dois tipos de requisitos: os funcionais e não funcionais.
Os requisitos funcionais tem como principal característica descrever o que o software fará, em um sentido objetivo. Um bom exemplo é: “O software deverá cadastrar funcionários”. Uma ação feita pelo software, sua funcionalidade.
Já os requisitos não funcionais tem como objetivo regrar um software, impor características nele, e definir processos do desenvolvimento. Por exemplo: “O software deverá permitir que apenas o gerente visualize os dados de todos funcionários” ou “O software será escrito na linguagem da programação C++”. 
Ambos tipos de requisitos tem suas subdivisões, os requisitos não funcionais tem mais delas por serem mais abrangentes. Ao compara-los, é possível perceber as diferenças. Os requisitos funcionais especificam a funcionalidade do software, e os não funcionais, as regras que os administram. Exemplos: [RF] O software deverá deletar usuário; [RNF] O software deve permitir que apenas administradores deletem usuários. Percebe a diferença? O Requisito Não Funcional (RFN) definiu uma regra para o Requisito Funcional (RF). Outra diferença dos requisitos não funcionais é que eles também especificam detalhes sobre o software, que não são considerados funções, como medidas de segurança, custos, desempenho, entre outros.
